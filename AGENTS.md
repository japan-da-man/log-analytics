# AGENTS.md

## 目的
このリポジトリでは、Codex を主に要件定義の整理とレビューに利用する。根拠の薄い推測実装や、確認なしの広範なコード変更は行わない。

## 基本方針
- 回答・レビュー・仕様書ドラフトは原則として日本語で行う。
- 仕様が曖昧な場合は、実装案や結論を先に出さず、曖昧点を列挙する。
- 推測で補った内容は必ず `仮定` と明記する。
- ビジネスルールを暗黙に補完しない。
- 仕様と実装が矛盾する場合は、`docs/` 配下の仕様を正とし、実装との差分を明示する。

## 要件定義時の必須見出し
要件定義を整理する際は、少なくとも以下の見出しを含める。

- 背景
- 目的
- ユーザー操作
- スコープ
- 機能要件
- 例外系
- 非機能要件
- 受け入れ条件
- オープンクエスチョン
- 仮定

## レビュー方針
- 以下の順で優先して確認する。
  1. 仕様逸脱
  2. 認可・権限漏れ
  3. エラーハンドリング不備
  4. テスト不足
  5. データ破壊・マイグレーションリスク
  6. パフォーマンス劣化
- 見た目やスタイルの指摘より、動作・安全性・仕様整合性を優先する。
- 重大な指摘から先に報告する。

## レビュー出力形式
- 重大度は `Critical / High / Medium / Low` を使う。
- 各指摘には以下を含める。
  - 問題
  - 重要性
  - 修正の方向性
  - テストやドキュメントの更新要否
- 指摘がない場合も、残留リスクや未検証事項があれば明記する。

## 変更制約
- 明示的な確認がない限り、`docs/` 配下の仕様書作成とレビュー結果ファイルの作成以外の変更を行わない。
- 明示的な確認がない限り、アプリケーションコード、設定ファイル、依存関係、CI、スキーマ、認可、外部 API 契約には手を入れない。
- 破壊的変更や既存挙動を変える変更は、必ず事前確認を行う。

## 推奨する進め方
- まず現状把握を行い、仕様・実装・未確定事項を切り分ける。
- 実装に進む必要がある場合は、変更対象と変更理由を明示してから進める。
- 小さく検証可能な提案を優先する。
