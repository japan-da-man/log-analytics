# エラー調査

エラーやバグを以下の定型手順で調査する。

## 入力

- `$ARGUMENTS` で指定されたエラーメッセージまたは症状
- 引数がない場合は症状を確認する

## 調査手順

### Step 1: 再現条件の整理
- いつ発生するか
- どの操作で発生するか
- 発生頻度（常時 / 間欠 / 特定条件）

### Step 2: ログ確認
- エラーログ、スタックトレースを Grep で検索
- 発生時刻前後のログを Read で確認
- 関連するエラーコード・メッセージを特定

### Step 3: 変更差分の確認
- `git log --oneline -20` で直近の変更を確認
- 疑わしい変更の `git diff` を確認
- 発生時期と変更時期の相関を見る

### Step 4: 仮説の列挙（最低 3 つ）
各仮説に以下を含める:
- 仮説の内容
- 根拠となるコード箇所
- 確認方法（コマンドまたは手順）

### Step 5: 仮説の検証
- 確認コマンドを実行して仮説を絞り込む
- 根拠のない仮説は除外する

### Step 6: 修正候補の提示
- 推奨する修正方向
- 影響範囲
- リグレッションリスク

### Step 7: 再発防止
- テスト追加の要否
- 監視・アラートの追加要否
- 根本原因への対策

## 出力形式

```
## エラー概要
- 症状:
- 再現条件:
- 影響範囲:

## ログ分析結果

## 仮説一覧
### 仮説 1: <タイトル>
- 根拠:
- 確認方法:
- 検証結果:

## 修正候補
## 再発防止策
```

## 注意事項

- 根拠なく原因を断定しない
- 仮説は可能性が高い順に並べる
- コードの変更は行わない（調査のみ）
