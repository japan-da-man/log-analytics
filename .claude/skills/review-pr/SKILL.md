# PR レビュー

変更差分を以下の固定観点でレビューする。

## 差分の取得

- 引数にブランチ名がある場合: `git diff main...<branch>`
- 引数に PR 番号がある場合: `gh pr diff <number>`
- 引数がない場合: `git diff main...HEAD`

## レビュー観点（この順序で確認する）

### 1. 仕様逸脱
- `docs/` の仕様と実装が一致しているか
- 要件の取りこぼしがないか

### 2. 例外処理
- エラーが握りつぶされていないか
- 失敗時のレスポンス・ログ・リトライが適切か

### 3. 認可漏れ
- 認証・認可ミドルウェアが適用されているか
- テナント境界・ロール境界が守られているか

### 4. テスト不足
- 新規コードに対応するテストがあるか
- 異常系・境界値のテストがあるか

### 5. 破壊的変更
- 既存の API 契約を壊していないか
- DB スキーマの互換性は保たれているか

### 6. 命名・責務崩れ
- 既存の命名規約から逸脱していないか
- 1 つの関数 / クラスに複数の責務が混在していないか

### 7. パフォーマンス懸念
- N+1 クエリ、不要な全件走査がないか
- リソースリーク（コネクション、ストリーム）がないか

## 出力形式

```
## 指摘事項

### <Critical|High|Medium|Low> <短いタイトル>
- 問題:
- なぜ重要か:
- 修正の方向性:

## 良い点
## 残留リスク
## 総評
```

## 注意事項

- 重大な指摘から順に報告する
- 軽微なスタイル指摘は最後にまとめる
- 指摘がない場合も残留リスクを明記する
