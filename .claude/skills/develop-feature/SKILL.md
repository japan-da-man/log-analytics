# 仕様書 → PR 作成 オーケストレーション

仕様書を受け取り、以下のフェーズを順に実行する。
各フェーズの完了条件を満たしてから次に進むこと。

## 入力

- 仕様書パス: `$ARGUMENTS`
- 引数がない場合は仕様書のパスを確認する

---

## Phase 1: 仕様分析

仕様書を読み、以下を整理する。

1. 仕様書を Read で読み込む
2. 以下を抽出・整理して出力する:
   - **目的**: この機能が解決する課題
   - **要件一覧**: 機能要件を番号付きで列挙
   - **受け入れ条件**: テスト可能な形で列挙
   - **影響範囲**: 変更が及ぶモジュール・ファイルの見立て
   - **未確定事項**: 仕様に曖昧な点があれば列挙
3. 未確定事項がある場合はユーザーに確認してから次へ進む

**完了条件**: 要件と受け入れ条件が明確になっていること

---

## Phase 2: 設計

`architect` サブエージェントに以下を依頼する。

- Phase 1 の分析結果と仕様書パスを渡す
- 既存コードのパターンを調査させる
- 以下を出力させる:
  - 変更対象ファイル一覧（新規 / 既存）
  - 責務分割とモジュール構成
  - インターフェース定義（型、スキーマ）
  - 依存関係の変更

API エンドポイントの設計が必要な場合は `/design-api` スキルの観点も含める。

**完了条件**: 変更対象ファイルと実装計画が明確になっていること

---

## Phase 3: テスト設計

`test-designer` サブエージェントに以下を依頼する。

- Phase 1 の受け入れ条件と Phase 2 の設計を渡す
- テストケース一覧（正常系・異常系・境界値）を出力させる
- モック戦略を決定させる

**完了条件**: implementer が着手できるテストケース一覧があること

---

## Phase 4: 実装

`implementer` サブエージェントに以下を依頼する。

- Phase 2 の設計と Phase 3 のテスト設計を渡す
- 実装順序:
  1. 型・スキーマ定義
  2. ビジネスロジック
  3. API ハンドラ / ルーティング（該当する場合）
  4. テストコード
- 小さく段階的に実装し、各段階でビルドが通ることを確認する
- 新規依存を追加する場合は `dependency-guardian` で妥当性を検証する

エンドポイント実装の場合は `/implement-endpoint` の手順に従う。

**完了条件**: コードが書かれていること

---

## Phase 5: 検証

Bash で以下を順に実行する。失敗した場合は原因を分析し、修正してから再実行する。

```
# 1. 型チェック（該当する場合）
npx tsc --noEmit

# 2. Lint（該当する場合）
npm run lint

# 3. テスト
npm test
```

検証コマンドはプロジェクトの実際の設定に合わせて調整する。
`package.json` の scripts を確認してから実行すること。

失敗時のループ:
1. エラーメッセージを分析する
2. 原因箇所を特定する
3. `implementer` で修正する
4. 再度検証する
5. 3 回失敗したらユーザーに報告して判断を仰ぐ

**完了条件**: lint / typecheck / test が全て通ること

---

## Phase 6: 自己レビュー

`code-reviewer` サブエージェントに以下を依頼する。

- `git diff main...HEAD`（または適切なベースブランチ）の差分を渡す
- `/review-pr` スキルの固定観点でレビューさせる:
  1. 仕様逸脱
  2. 例外処理漏れ
  3. 認可漏れ
  4. テスト不足
  5. 破壊的変更
  6. 命名・責務崩れ
  7. パフォーマンス懸念
- Critical / High の指摘がある場合は Phase 4 に戻って修正する

**完了条件**: Critical / High の指摘が解消されていること

---

## Phase 7: PR 作成準備

以下を出力する。

### 7a. 差分要約
- 変更ファイル一覧と各ファイルの変更概要

### 7b. PR タイトル・本文
`/commit-message pr` の形式で:

```
## 概要
<目的と変更の要約>

## 変更内容
- <主要な変更を箇条書き>

## テスト
- <テスト結果サマリ>
- <テスト観点>

## 影響範囲
- <影響を受けるコンポーネント>

## 自己レビュー結果
- <残留リスク・Medium/Low の指摘事項>

## 関連
- <仕様書パス>
```

### 7c. 残課題
- Phase 6 で Medium / Low だった指摘事項
- 今後対応が必要な技術的負債
- フォローアップが必要な項目

**完了条件**: PR タイトル案と本文案が出力されていること

---

## 制約（全フェーズ共通）

- 仕様にないビジネスルールは追加しない
- 仮定が入る場合は「仮定」と明記する
- 既存の命名規約・責務分離・テスト方針を優先する
- 破壊的変更がある場合は必ずユーザーに確認する
- `docs/` 配下の仕様書を変更しない
