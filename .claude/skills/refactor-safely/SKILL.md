# 安全なリファクタリング

リファクタリングを以下の手順で安全に進める。

## 入力

- `$ARGUMENTS` で指定された対象ファイルまたはモジュール
- 引数がない場合は対象を確認する

## 手順

### Step 1: 目的の確認
- なぜリファクタするのか明確にする
- 期待する改善（可読性、保守性、パフォーマンスなど）を言語化する

### Step 2: 変更範囲の特定
- 対象ファイル一覧を列挙する
- Grep で対象の参照箇所を全て洗い出す
- 影響を受ける呼び出し元を特定する

### Step 3: Public Interface の維持確認
- エクスポートされている関数・型・定数を列挙する
- 変更後もシグネチャが維持されるか確認する
- 破壊的変更がある場合は事前に報告する

### Step 4: テストの確認と追加
- 既存テストがリファクタ対象をカバーしているか確認する
- カバレッジが不足している場合はリファクタ前にテストを追加する
- テストが通ることを確認してからリファクタに着手する

### Step 5: リファクタの実施
- 小さく段階的に変更する
- 各段階でテストが通ることを確認する

### Step 6: Before / After の確認
- `git diff` で変更内容を確認する
- 機能的な変更が混入していないか確認する
- テストが全て通ることを最終確認する

## 出力形式

```
## リファクタ目的
## 変更範囲
## Public Interface への影響
## テスト状況（before）
## 変更内容
## テスト結果（after）
## 残課題
```

## 注意事項

- 機能変更とリファクタを混ぜない
- テストが不足している状態でリファクタを始めない
- public interface を壊す場合は必ず事前確認する
