# API 設計

新しい API エンドポイントの設計を以下の観点で整理する。

## 入力

- `$ARGUMENTS` で指定されたエンドポイントまたは機能名
- `docs/` 配下の関連仕様書
- 既存の API 実装パターン

引数がない場合は、対象のエンドポイントまたは機能名を確認する。

## 設計手順

1. `docs/` の仕様書から要件を確認する
2. 既存 API のパターン（ルーティング、命名、エラー形式）を Grep / Read で把握する
3. 以下のテンプレートに沿って設計を出力する

## 出力テンプレート

```
## エンドポイント定義
- メソッド:
- パス:
- 概要:

## 認可
- 必要な権限 / ロール:
- 未認可時のレスポンス:

## Request
### パラメータ（path / query / header）
### Request Body（JSON Schema or Zod）

## Response
### 成功時（status code + body）
### エラー時（status code + エラー形式）

## バリデーション
- 各フィールドの制約:
- 入力不正時の挙動:

## DB 影響
- 読み取り対象テーブル:
- 書き込み対象テーブル:
- マイグレーション要否:

## テスト観点
- 正常系:
- 認可エラー:
- バリデーションエラー:
- 境界値:
- 冪等性:

## 未確定事項
```

## 注意事項

- 既存の API 規約（エラー形式、命名、認可パターン）に合わせる
- 仕様が曖昧な箇所は推測で埋めず「未確定事項」に列挙する
