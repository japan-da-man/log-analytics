# Requirements Refiner

要求が曖昧な状態から、開発着手可能な要件定義のたたき台へ落とし込むための skill。

## Use when

- 要件定義をしたい
- 仕様がふわっとしているので整理したい
- アイデアを仕様書の形にしたい
- スコープや対象外を明確にしたい
- 実装前に抜け漏れを洗いたい

## Do first

1. 依頼文、既存 docs、関連 issue を確認する。
2. 情報が不足している場合は、曖昧点を先に列挙する。
3. 推測で補った内容は必ず `仮定` として明示する。
4. ビジネスルールを暗黙に補完しない。

## Required checks

要件整理の前に、少なくとも次を確認する。

- 何のために作るのか
- 誰が使うのか
- ユーザーが何を操作するのか
- 何を今回の対象外にするのか
- 成功と失敗をどう判定するのか
- エラー時や例外時に何が起きるべきか
- 性能、可用性、監査、セキュリティなどの非機能条件はあるか

## Output format

以下の見出しを固定で使う。

```md
# <機能名>

## 背景
## 目的
## ユーザーストーリー
## 対象範囲
## 対象外
## 機能要件
## 非機能要件
## 例外系
## 受け入れ条件
## 未確定事項
## 仮定
```

## Writing guidance

- `背景` は現状の課題と文脈を書く。
- `目的` は達成したい結果を書く。
- `ユーザーストーリー` は利用者、行動、期待価値を書く。
- `対象範囲` は今回対応する内容だけを書く。
- `対象外` は誤解されやすいが今回はやらない内容を書く。
- `機能要件` は番号付きで、観測可能な振る舞いとして書く。
- `非機能要件` は性能、運用、監査、可観測性、セキュリティを含めて検討する。
- `例外系` は入力不正、権限不足、依存先障害、重複操作、タイムアウトを優先して確認する。
- `受け入れ条件` は曖昧な表現を避け、テスト可能な文にする。
- `未確定事項` は、意思決定が必要な論点を列挙する。

## Quality bar

- 説明文で終わらせず、実装やレビューに使える粒度まで具体化する。
- `いい感じに`, `適切に`, `必要に応じて` のような曖昧語は避ける。
- 仕様書の読み手が PM、設計者、実装者、レビュアーでも解釈がぶれにくい形を目指す。
- 既存仕様と衝突する場合は、差分を明示する。
