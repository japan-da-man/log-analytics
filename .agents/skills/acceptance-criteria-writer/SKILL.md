# Acceptance Criteria Writer

要件を、テスト可能でレビュー可能な受け入れ条件へ変換するための skill。

## Use when

- 受け入れ条件を書きたい
- 要件はあるがテスト観点が弱い
- Given / When / Then に落としたい
- 成功条件、失敗条件、境界値を整理したい

## Inputs

次のいずれかを入力として扱う。

- 要件定義書
- 機能一覧
- ユーザーストーリー
- 仕様メモ

入力が不足している場合は、受け入れ条件を書き始める前に不足点を明示する。

## Workflow

1. 各要件を独立した振る舞いに分解する。
2. 正常系、異常系、境界値、権限差分を分けて考える。
3. 観測可能な結果に変換する。
4. 必要ならログ、監査、メトリクスの観点を追加する。
5. セキュリティ条件がある機能では、権限、入力検証、情報漏えい観点を必ず含める。

## Output format

要件ごとに次の形で出力する。

```md
## 要件 <ID or title>

### Given / When / Then
- Given:
- When:
- Then:

### 成功条件
- ...

### 失敗条件
- ...

### 境界値
- ...

### ログ / メトリクス観点
- ...

### セキュリティ観点
- ...
```

## Writing guidance

- `Given` は前提状態、権限、入力状態を明確にする。
- `When` はユーザー操作、API 呼び出し、イベント発火を明確にする。
- `Then` は画面表示、レスポンス、DB 更新、通知、副作用を観測可能な形で書く。
- 成功条件には、完了条件と永続化内容を含める。
- 失敗条件には、エラーコード、メッセージ、再試行可否、副作用なしを含める。
- 境界値には、最小、最大、空、重複、期限切れ、同時実行を含める。
- ログ / メトリクス観点には、運用時に確認すべきイベントや失敗率を含める。
- セキュリティ観点には、認可、PII、監査証跡、入力検証、レート制限の必要性を含める。

## Quality bar

- 受け入れ条件は、QA や自動テストに直接変換しやすい粒度で書く。
- UI と API のどちらが起点でも、結果が一致するか確認する。
- 曖昧な言葉ではなく、期待結果を具体的に書く。
- 単なる要件の言い換えで終わらせない。
